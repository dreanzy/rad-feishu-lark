# AGENTS.md

1. 用户是非技术产品经理，回答尽量少讲底层技术，多从交互、业务流、代码逻辑解释。
2. 本项目只是 Pi 和飞书/Lark 的消息桥，不是任务系统。
3. 飞书扩展只负责：接收飞书消息、发送 Pi 回复、把 Pi 的关键结果回传到原飞书会话。
4. 飞书扩展可以保存轻量路由信息，例如 `sessionKey -> chatId/topicId`。
5. 飞书扩展可以保存 Pi 内部任务到飞书来源的映射，例如 `jobId -> chatId/topicId`。
6. 飞书扩展不要自己实现定时器、提醒、cron、重复任务、长任务执行器。
7. 飞书扩展不要复制 Pi 内部能力，不要和 Pi 内部工具竞争。
8. 定时任务、重复任务、长任务、自动化、工具调用，都应该走 Pi 内部能力。
9. 飞书扩展只监听 Pi 的关键结果，再决定是否发回飞书。
10. 只有从飞书创建的 Pi 内部任务，完成/失败/需要用户处理时才回传飞书。
11. 从 TUI、CLI 或其他渠道创建的任务，不要主动发到飞书。
12. 不要把搜索过程、工具调用过程、调试日志、中间状态主动发到飞书。
13. 可以回传：任务完成、任务失败、定时提醒结果、需要用户确认的信息。
14. 持久化只存桥接和投递信息，不存任务本体。
15. 修改前先判断：这是飞书传输/路由问题，还是 Pi 产品能力问题？
16. 如果是 Pi 产品能力，不要写进飞书扩展。
